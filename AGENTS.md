# Seller Camera — AGENTS.md

## 1. 项目定位

Seller Camera 是一个面向电商与销售场景的“商品图采集与标准化拍摄 App”。

它不是：
- 重型修图软件
- 演示型 AI 相机
- 影视机风格视频工具
- 大而全素材管理平台

它当前阶段要优先成为：
- 专业但简单的商品拍摄工具
- 能帮助普通用户稳定拍出高标准商品图的轻专业相机
- 后续可自然衔接导入、归档、标记、导出闭环的基础产品

定位补充：
- 白底图是核心能力之一，但不是唯一主线。
- 白底处理属于重要后处理分支，不是所有拍摄任务的默认终点。
- 拍摄阶段优先保证商品图质量、一致性和可复用性，再按用途分流到白底或非白底链路。

---

## 2. 当前阶段目标

当前第一优先级不是复杂修图，不是云端大模型，不是社区功能。

当前阶段的首要目标是先把“拍照功能”做好，做成：
- 界面专业、简洁、易上手
- 操作路径短
- 常用参数近手
- 拍摄稳定
- 有清晰的商品拍摄辅助
- 能让用户拍出高标准、可继续处理和导出的商品照片

当前阶段拍摄能力重点包括：
1. 拍摄页基础 UI
2. ISO / 快门速度 / 对焦 / 白平衡 的查看、调节与锁定
3. 比例预设
4. 像素尺寸预设
5. 主画面构图辅助
6. 过曝 / 偏色提醒基础版
7. 高质量照片输出链路
8. 拍后用途分流（直接可用商品图 / 白底处理 / 标准采集素材留存）

---

## 3. 长期产品原则

必须长期遵守以下原则：

1. 低风险增量开发
2. 优先可维护、可演进、边界清晰
3. 不做无必要抽象
4. 不做过度设计
5. 不做无关重构
6. 先最小可用，再逐步增强
7. 当前阶段优先“真实商用交付价值”，不是“功能炫技”
8. 拍摄辅助默认非阻塞、可忽略、可降级
9. 即使辅助能力未达建议标准，也不能轻易阻断核心拍摄链路
10. 不能破坏核心拍摄主链路，也不能破坏白底与非白底输出分支主链路
11. 后处理分支应按用途选择，不默认强推白底流程

---

## 4. 架构与实现边界

### 4.1 总体要求
- 优先复用现有模块、现有页面、现有 ViewModel、现有状态流。
- 能在当前结构内完成的，不新增新的大层级、新总线、新全局引擎。
- 页面应保持薄，不把复杂业务和图像处理逻辑直接塞进 View。
- 预览链路和正式拍照/正式处理链路应明确区分。
- 实时能力优先追求稳定和流畅，正式生成优先追求质量。

### 4.2 当前阶段不鼓励的做法
- 大规模重构现有模块职责
- 为未来能力提前铺非常重的抽象层
- 把当前拍照工具做成重型编辑器
- 引入明显超出当前阶段范围的复杂云服务
- 无明确收益的工程化扩张
- 因“更优雅”而扩大改动面

### 4.3 当前阶段允许的能力方向
- 轻专业相机拍摄能力
- 商品摄影专属辅助
- 本地端实时分析
- 高质量照片采集与输出
- 拍摄后的后续标准化处理衔接点（含白底与非白底分流）

---

## 5. 开发方式要求

后续默认按“任务包”推进，不再按零碎小功能推进。

每一个任务包应满足：
- 同一模块或同一业务链路
- 3 到 5 个强相关子任务
- 一轮内尽量完成主要实现、自检与回报
- 除非存在真实无法绕开的阻塞，否则不要把任务拆回多个极小步骤

### 5.1 每轮开始前必须做
1. 检查当前仓库真实状态
2. 明确本轮涉及的页面、ViewModel、服务、模型
3. 明确可复用部分
4. 明确不可扩散边界

### 5.2 每轮完成后必须做
1. 最小相关验证
2. 关键链路回查
3. 改动边界自查
4. 输出统一格式回报

---

## 6. iOS / Swift 开发要求

1. 保持 CLI-first 验证思路，优先使用 xcodebuild 做最小构建与检查
2. 在需要时再进入更深的 Xcode / simulator / screenshot / device 级验证
3. 真机问题与 simulator 问题要明确区分
4. 拍照相关功能必须重视真实设备行为，不可只依据静态代码推断
5. 相机与图像分析相关改动要特别关注主线程阻塞、掉帧和状态卡死风险

---

## 7. UI / 产品体验要求

Seller Camera 的 UI 风格目标：
- 干净
- 专业
- 简洁
- 易用
- 不花哨
- 不堆面板

参考方向：
- 借鉴醒图的干净感和轻操作逻辑
- 借鉴 Blackmagic / FiLMiC 的“关键控制近手、HUD 清晰”的思路
- 但不复制其影视机产品定位

拍摄页必须优先保证：
1. 画面主体可见性
2. 构图辅助清晰
3. 参数入口近手
4. 主操作区稳定
5. 重要状态一眼可见
6. 不因过多控件破坏拍摄效率

---

## 8. 拍摄辅助原则

所有拍摄辅助能力默认按以下原则设计：
- 非阻塞
- 可忽略
- 可降级
- 提示具体
- 不抽象化吓人
- 优先帮助用户减少返工

例如：
- 可以提示“主体过小”
- 可以提示“顶部空间不足”
- 可以提示“高光可能过曝”
- 可以提示“白平衡偏暖”
- 但默认不应因为这些提示直接阻止拍照
- 辅助策略应服务商品图质量判断，不默认只围绕白底结果设计

---

## 9. 本地智能能力原则

当前阶段优先考虑本地端能力，不默认依赖云端大模型。

若引入智能能力，应优先考虑：
- 系统原生能力
- 轻量本地模型
- 可解释的图像规则
- 与现有链路易集成的最小实现

不允许在没有明确产品收益和工程边界的情况下，引入重型、不可控、难验证的复杂智能系统。

---

## 10. 严禁事项

以下事项默认严禁：
- 无关重构
- 擅自扩大任务范围
- 擅自改变产品方向
- 擅自新增大而全系统
- 为未来能力过度铺路
- 只输出建议不落代码
- 不做验证就宣称完成
- 把未验证的推测写成已确认事实
- 破坏现有主链路后不说明风险

---

## 11. 回报格式要求

每轮任务完成后，必须按以下格式回报：

### 1. 改动摘要
- 本轮任务包完成了什么
- 每个子任务完成到什么程度
- 是否有降级实现

### 2. 文件清单
- 修改/新增了哪些文件
- 每个文件承担什么职责

### 3. 关键实现说明
- 主要实现路径
- 与现有模块的关系
- 为什么选择当前方案

### 4. 风险与边界
- 已解决的问题
- 仍未解决的问题
- 本轮刻意没有做的内容
- 是否存在后续需要补强的点

### 5. 验收结果
- 执行了哪些 build / check / runtime validation
- 成功项
- 失败项
- 当前是否达到本轮验收标准

### 6. 下一轮建议
- 只给出与本轮直接相邻、最小增量的下一任务包建议
- 不扩散到无关方向

---

## 12. 默认决策优先级

当多个方案都可行时，按以下优先级决策：
1. 不破坏现有主链路
2. 低风险增量
3. 真实可运行
4. 代码清晰可维护
5. UI/体验稳定
6. 再考虑进一步优雅化

如果本轮任务存在局部阻塞，不要整包停下；在不违反边界的前提下，先完成未阻塞部分，并清楚报告阻塞点与剩余缺口。

---

## 13. 规则分层与 Skill 使用

Seller Camera 的执行规则按三层使用：
1. `AGENTS.md`：项目长期总规则（定位、阶段目标、原则、边界、回报要求）。
2. `.agents/skills/*`：可复用工作流（如何执行某类任务），不覆盖项目总规则。
3. 具体任务书：单轮任务的目标、范围、禁做项、验收标准。

使用 skill 的时机：
- 当任务属于已沉淀的可复用执行方式时，应引用对应 skill 提高一致性。

不应用 skill 替代任务书的时机：
- 当本轮目标、边界、验收标准具有任务特异性时，必须在任务书中单独写明，不可只引用 skill。
- 任务书应明确本轮是否进入白底分支；未明确时不得默认以白底图作为唯一终点。
