# Seller Camera BiRefNet 替代桥接路线评估（R05）

## 1) 为什么现在进入替代桥接论证

R01-R04 证据链已完整闭环：
1. R01：审查未通过（缺模型资产与最小桥接）。  
2. R02：provider 路由与最小桥接骨架已成立。  
3. R03：ONNX 资产已落地，但 CoreML 成品资产未形成，单点闭环失败于 `mlmodelc_not_found`。  
4. R04：三条 CoreML 最小转换路径继续失败，单点闭环仍失败于模型加载层。  

当前结论不是“BiRefNet 永久不可行”，而是“CoreML 打通继续同类试错边际收益低”。  
因此本轮进入路线级判断，而不是继续开 CoreML 局部试错包。

---

## 2) 本轮替代桥接评估范围

### 2.1 评估对象（本轮只评估）
- ONNX Runtime iOS 路线（含 CoreML EP/缓存能力作为可选执行策略）。

### 2.2 明确不在本轮范围
- ONNX Runtime 正式接入实现  
- BiRefNet admission 跑测  
- 白底主流程/下游联动改造  
- 多候选并行桥接  
- 云端推理路线

---

## 3) ONNX Runtime 路线工程评估

## 3.1 工程边界适配性
1. 可以继续收敛在 `SegmentationProvider`，保持 Vision 默认路径。  
2. 不需要改 `segmentation -> refine -> decontam -> compose` 主顺序。  
3. 可保留现有失败回退语义（模型加载失败/推理失败回退 Vision）。

结论：**边界上可控，但前提是严格禁止越界扩散。**

## 3.2 最小可行性
1. iOS 侧存在现实集成路径（依赖 + provider 最小调用）。  
2. 可定义清晰最小闭环：`load -> infer -> mask -> fallback`。  
3. 可在不进入 admission 的情况下先做单点验证。

结论：**具备进入“最小可行化包”所需的技术入口。**

## 3.3 成本
1. 新依赖引入与工程集成成本：中到高。  
2. 二进制体积与构建复杂度成本：中到高。  
3. 后续维护与排障成本：中到高。  

## 3.4 风险
1. 运行时兼容与真机行为存在不确定性。  
2. 调试复杂度高于当前 Vision/CoreML 直连路径。  
3. 若边界管理失效，容易引发运行时体系扩张风险。

## 3.5 收益
1. 有现实机会绕开当前 CoreML 成品资产阻塞。  
2. 有机会把 BiRefNet 推进到“可进入 admission R01”的工程前置条件。  
3. 相比继续 CoreML 同类试错，新增信息增量更高。

---

## 4) 是否符合 Seller Camera 当前项目原则

在以下硬约束下，**可以条件性符合**：
1. 只做 provider 边界内最小可行化，不改主链。  
2. 只做单点闭环，不做 admission 样本跑测。  
3. 保留 Vision 默认路径和失败回退。  
4. 不引入第二候选并行，不做大规模运行时抽象。  

若任一约束失守，则不再符合“低风险增量”原则。

---

## 5) 二选一结论

**结论：值得进入“BiRefNet 替代桥接路线最小可行化包”。**

理由（硬结论）：
1. CoreML 路线在当前边界已连续失败，继续同类试错边际收益低。  
2. ONNX Runtime 路线存在现实工程入口，且有机会直接解除当前唯一硬阻塞（CoreML 成品资产不可得）。  
3. 在严格边界约束下，风险可控且可回退。  

这不是“BiRefNet 可进 admission”的结论，只是“值得开下一包做最小可行化验证”的结论。

---

## 6) 下一包最小边界（仅在本结论下）

### 允许做
1. 最小依赖引入（仅服务 ORT 单点闭环）。  
2. `SegmentationProvider` 内最小 ORT provider 对接。  
3. 单点 `load -> infer -> mask` 验证。  
4. 失败回退验证。  

### 不允许做
1. admission_v1 跑测。  
2. 主链重写。  
3. refinement / 去污染 / 合成联动改造。  
4. 并行引入第二候选。  

### 下一包通过标准
1. 可真实加载模型。  
2. 可触发真实推理。  
3. mask 可回到 provider 边界。  
4. Vision 回退保持可用。  

未满足以上任一项则继续 `stop_current_candidate`。
